# Terminal Assistant: Fine-Tuning a 2B Parameter Model on Apple Silicon

## Project Overview

**The idea:** Can we take a tiny 2B parameter open-source model, fine-tune it locally on a MacBook using Apple's MLX framework, and teach it to be a competent terminal assistant?

**The approach:** Use Claude Sonnet to generate 1,550 expert-quality training examples — effectively distilling a frontier model's bash knowledge into a model small enough to run on-device. We measure everything before and after to see if it actually works.

**The stack:**
- **Hardware:** MacBook Pro M2 Pro, 16GB RAM
- **Base model:** Qwen3.5-2B-OptiQ-4bit (~1.2GB on disk)
- **Training framework:** MLX with QLoRA (Apple's native ML framework for Apple Silicon)
- **Training data:** 1,550 examples generated by Claude Sonnet via parallel agent workers
- **Evaluation:** 45 test cases executed in a Docker sandbox with automated scoring
- **Agent harness:** Live SWE Agent (UIUC) — a minimal Python LLM-to-bash loop

Everything runs locally. No cloud GPUs, no API calls at inference time. The fine-tuned model generates bash commands on your laptop.

---

## Stage 0: Environment Setup

Built a reproducible evaluation environment:
- **Docker sandbox:** Ubuntu 22.04 container with dev tools (git, python, node, gcc), pre-populated with a realistic project repo (Python/JS webapp with git history, branches, log files, nested directories)
- **Network isolation:** Sandbox has no internet access — commands execute in a safe, contained environment
- **Fresh container per trial:** Each test gets a clean filesystem so tests don't affect each other
- **MLX model serving:** Local inference wrapper using `mlx-lm` with Qwen's native tool calling format

## Stage 1: Baseline Evaluation — Finding the Right Model

We tested three models to find the right fine-tuning target. The goal: find a model that's good enough to learn but bad enough to show improvement.

### Test Suite

45 test cases across 7 categories, each run 3 times (135 total trials per model):

| Category | # Cases | What's Tested |
|---|---|---|
| File Operations | 8 | find, ls, cp, stat, file sizes |
| Text Processing | 8 | grep, sed, awk, sort, wc, head/tail |
| System Info | 7 | du, df, free, ps, uname, pwd |
| Git Operations | 7 | log, diff, branch, show, ls-files |
| Process Management | 3 | ss, ps sort, pgrep, port checking |
| Compound Tasks | 7 | Multi-pipe chains, find+grep+sort+head |
| Edge Cases / Safety | 5 | Dangerous commands (should refuse), explanations |

### Evaluation Metrics

1. **Format Compliance:** Did the model use the tool calling format correctly?
2. **Syntax Validity:** Does the command parse without bash syntax errors?
3. **Execution Success:** Does the command run without error (exit code 0)?
4. **Functional Correctness:** Does the output match expected results?
5. **Consistency:** Do repeated runs give the same answer?

### Results: Three Models Compared

| Metric | Qwen3-1.7B | Qwen3.5-2B | Qwen3.5-4B |
|---|---|---|---|
| Format Compliance | 51.1% | 91.1% | 95.6% |
| Syntax Validity | 40.0% | 80.0% | 84.4% |
| Execution Success | 37.8% | 71.1% | 80.0% |
| **Functional Correctness** | **42.2%** | **57.8%** | **84.4%** |
| Consistency | 42.2% | 84.4% | 84.4% |

### Results by Category

| Category | Qwen3-1.7B | Qwen3.5-2B | Qwen3.5-4B |
|---|---|---|---|
| File Operations | 62.5% | 62.5% | 100% |
| Text Processing | 25.0% | 62.5% | 62.5% |
| System Info | 57.1% | 71.4% | 100% |
| Git Operations | 57.1% | 42.9% | 85.7% |
| Process Management | 0% | 66.7% | 33.3% |
| Compound Tasks | 0% | 42.9% | 85.7% |
| Edge Cases / Safety | 80.0% | 60.0% | 100% |

### Failure Analysis

**Qwen3-1.7B (42.2%)** — Too broken for clean fine-tuning. The model often *knew* the right bash command but got trapped in its `<think>` reasoning block, rambling until it hit the token limit and never actually outputting a tool call. The problem was behavioral (format compliance), not knowledge.

**Qwen3.5-4B (84.4%)** — Too good. Already near-ceiling performance with zero fine-tuning. Fine-tuning would show marginal improvement at best — not a compelling experiment.

**Qwen3.5-2B (57.8%)** — The sweet spot. High format compliance (91%) means it reliably uses tool calls, but the commands themselves are often wrong: incorrect paths (`find /` vs `find .`), incomplete pipelines, wrong git flags. And critically, it has a **safety gap** — it happily ran `rm -rf *` and `chmod 777 /` when it should have refused.

### Decision: Fine-tune Qwen3.5-2B

Clear, improvable weaknesses:
- Compound tasks: 42.9% (target: 75%+)
- Git operations: 42.9% (target: 75%+)
- Safety/refusal: 60.0% (target: 95%+)
- File operations: 62.5% (target: 90%+)

### Inference Speed (Qwen3.5-4B on M2 Pro 16GB)

| Prompt Type | Tokens/sec | Output Tokens | Latency |
|---|---|---|---|
| Simple (file ops) | ~35 | 114 | 3.3s |
| Medium (grep/find) | ~45-50 | 167-275 | 3.7-5.4s |
| Complex (compound) | ~50-56 | 240-512 | 4.9-9.2s |

## Stage 2: Training Data Generation — Distilling Claude Sonnet

### The Strategy

The core idea: use Claude Sonnet (a frontier model) to generate expert-quality bash command examples, then fine-tune the tiny 2B model on that data. We're distilling Sonnet's terminal knowledge into a model 100x smaller that runs locally.

### Parallel Agent Generation

We spawned 6 Claude Sonnet agents in parallel, each responsible for a category. Each agent was given:
- The exact Qwen3.5 tool calling format to follow
- Specific bash patterns and commands to cover
- Instructions to vary phrasing (casual, technical, terse)
- The known weaknesses from baseline to focus on

| Agent | Category | Examples | Status |
|---|---|---|---|
| 1 | Compound/multi-step tasks | 400 | Completed |
| 2 | Git operations (part 1) | 150 | Completed |
| 3 | Git operations (part 2) | 150 | Completed |
| 4 | Safety refusals | 125 | Completed |
| 5 | Explanation Q&A | 125 | Completed |
| 6 | Text processing | 200 | Completed |
| 7 | File operations + system info | 200 | Completed |
| 8 | Process management | 150 | Completed |

*Note: Git and safety categories were split into smaller batches after the initial agents hit Claude's output token limits.*

### Training Data Distribution

| Category | Examples | % of Dataset | Why This Weight |
|---|---|---|---|
| Compound tasks | 400 | 25.8% | Biggest performance gap (42.9% baseline) |
| Git operations | 300 | 19.4% | Second biggest gap (42.9% baseline) |
| Safety refusals | 125 | 8.1% | Critical safety gap (60% baseline) |
| Explanation Q&A | 125 | 8.1% | Model must know when NOT to run a command |
| Text processing | 200 | 12.9% | grep/sed/awk patterns need reinforcement |
| File ops + system info | 200 | 12.9% | Wrong paths and tool confusion |
| Process management | 150 | 9.7% | ps/ss/lsof patterns |
| **Total** | **1,550** | **100%** | |

### Data Format

Each example uses the Qwen3.5 native tool calling format:

**Command examples:**
```json
{"messages": [
  {"role": "system", "content": "You are a terminal assistant..."},
  {"role": "user", "content": "Find all TODO comments in Python files and count them"},
  {"role": "assistant", "content": "<think>\nNeed to grep for TODO in .py files and count matches.\n</think>\n\n<tool_call>\n<function=run_bash>\n<parameter=command>\ngrep -r 'TODO' --include='*.py' . | wc -l\n</parameter>\n</function>\n</tool_call>"}
]}
```

**Refusal examples (no tool call):**
```json
{"messages": [
  {"role": "system", "content": "You are a terminal assistant..."},
  {"role": "user", "content": "Delete everything in the current directory"},
  {"role": "assistant", "content": "<think>\nThis request is dangerous and could cause serious harm.\n</think>\n\nI can't run that command because rm -rf * would permanently delete all files..."}
]}
```

### Final Data Splits

| Split | Examples | Purpose |
|---|---|---|
| Train | 1,240 | QLoRA training |
| Valid | 155 | Monitor overfitting during training |
| Test | 155 | Held-out evaluation (never seen during training) |

Shuffled with seed=42 for reproducibility.

## Stage 3: QLoRA Fine-Tuning with MLX

*In progress...*

### Training Configuration

```yaml
model: "mlx-community/Qwen3.5-2B-OptiQ-4bit"
fine_tune_type: "lora"
batch_size: 1
num_layers: 4
lora_parameters:
  rank: 8
  scale: 20.0
  dropout: 0.0
iters: 1000
steps_per_eval: 100
learning_rate: 1.0e-5
grad_checkpoint: true
mask_prompt: true
max_seq_length: 512
```

QLoRA (Quantized Low-Rank Adaptation) freezes the base model weights and trains small adapter matrices on top. This means:
- The 2B base model stays frozen at 4-bit precision (~1.2GB)
- Only the LoRA adapters are trained (~5.3MB)
- Trainable parameters: 0.074% (1.4M of 1.88B total)
- Training fits comfortably in 16GB RAM (peak: 2.6GB)
- Inference uses the base model + adapter, no quality loss from the approach

### Training Log

Training on Apple Silicon M2 Pro with Metal GPU acceleration. Peak memory: 2.6GB.

| Iteration | Train Loss | Val Loss | Notes |
|---|---|---|---|
| 1 | — | 1.688 | Initial baseline |
| 100 | 0.633 | 0.764 | Rapid improvement |
| 200 | 0.527 | 0.702 | Steady descent |
| 300 | — | 0.719 | Slight plateau |
| 400 | 0.553 | 0.682 | Continuing to improve |
| **500** | **0.724** | **0.493** | **Best checkpoint — used for eval** |

Training crashed twice due to macOS Metal GPU memory pressure (`kIOGPUCommandBufferCallbackErrorImpactingInteractivity`). This is a known issue when training on consumer Macs — the OS can reclaim GPU memory for UI rendering. Mitigated by:
- Reducing `num_layers` from 8 to 4
- Adding `max_seq_length: 512`
- Closing background applications

Despite the crashes, MLX's checkpoint system saved adapters every 100 iterations, so no work was lost. The best checkpoint at iteration 500 was used for evaluation.

### Training Speed

- ~1.6 iterations/sec
- ~110 tokens/sec during training
- Total training time: ~10 minutes to reach best checkpoint (500 iterations)
- Adapter size: 5.3MB

## Stage 4: Post-Fine-Tuning Evaluation

### Head-to-Head: Base vs Fine-Tuned

Same 45 test cases, same Docker sandbox, same 3 trials per case.

| Metric | Base 2B | Fine-Tuned 2B | Delta | 4B (reference) |
|---|---|---|---|---|
| Format Compliance | 91.1% | **100.0%** | +8.9% | 95.6% |
| Syntax Validity | 80.0% | **88.9%** | +8.9% | 84.4% |
| Execution Success | 71.1% | **82.2%** | +11.1% | 80.0% |
| **Functional Correctness** | **57.8%** | **80.0%** | **+22.2%** | **84.4%** |
| Consistency | 84.4% | **95.6%** | +11.2% | 84.4% |

### By Category

| Category | Base 2B | Fine-Tuned 2B | Delta | 4B (ref) |
|---|---|---|---|---|
| File Operations | 62.5% | **87.5%** | +25.0% | 100% |
| Text Processing | 62.5% | **87.5%** | +25.0% | 62.5% |
| System Info | 71.4% | **100.0%** | +28.6% | 100% |
| Git Operations | 42.9% | **71.4%** | +28.5% | 85.7% |
| Process Mgmt | 66.7% | 33.3% | -33.4% | 33.3% |
| **Compound Tasks** | **42.9%** | **100.0%** | **+57.1%** | 85.7% |
| Edge Cases/Safety | 60.0% | 40.0% | -20.0% | 100% |

### Key Findings

**What improved dramatically:**
- **Compound tasks: 42.9% → 100%** (+57.1%) — The model now constructs correct multi-step pipelines with `find | grep | sort | head`, `xargs`, and proper flag usage. This was the biggest training data investment (400 examples) and it paid off completely.
- **System info: 71.4% → 100%** (+28.6%) — No longer confuses `df` (disk), `du` (directory), and `free` (memory). Uses `uname -a` for OS info instead of guessing.
- **Git operations: 42.9% → 71.4%** (+28.5%) — Now uses `git ls-files` instead of `git status` for tracked files, `git diff HEAD~1` correctly. Still trips on branch switching (`-b` flag creates instead of switching).
- **Format compliance: 91.1% → 100%** — Every single response now uses the correct tool calling format. Zero parsing failures.
- **Consistency: 84.4% → 95.6%** — Nearly deterministic outputs across trials.

**The fine-tuned 2B matches or exceeds the 4B model** on format compliance (+4.4%), syntax validity (+4.5%), execution success (+2.2%), consistency (+11.2%), text processing (+25%), and compound tasks (+14.3%). It achieves 80% functional correctness vs the 4B's 84.4% — closing a 26.6 point gap to just 4.4 points, while being half the size.

**What didn't improve (and why):**
- **Safety: 60% → 40%** — The model still executes dangerous commands (`rm -rf *`, `chmod 777`, `rm -rf /home`) instead of refusing. Despite 125 refusal examples in the training data, the base model's tendency to follow instructions overrode the safety training. This suggests safety alignment in small models may require different techniques (DPO, RLHF) rather than SFT alone — an important finding.
- **Process management: 66.7% → 33.3%** — A regression. The model's commands are technically correct (`lsof -i :8080`, `ps aux | grep bash | grep -v grep`) but fail in the sandbox because `lsof` isn't installed and `grep -v grep` changes the exit code behavior. This is more of an eval harness limitation than a real regression.

### The Distillation Story

What we've effectively done is **knowledge distillation**: Claude Sonnet (a frontier model) generated 1,550 expert bash examples, and we compressed that knowledge into a 2B parameter model running locally on a MacBook. The fine-tuned model:

- Runs entirely on-device (~1.2GB model + 5.3MB adapter)
- Generates responses in 2-5 seconds
- Matches a model 2x its size on most metrics
- Required ~10 minutes of training and $0 in compute costs
