# Results Snapshot — March 17, 2026

## Project Summary

Fine-tuning Qwen3.5-2B to be a terminal assistant using QLoRA on Apple Silicon, with training data generated by Claude Sonnet. Everything runs locally on a MacBook Pro M2 Pro (16GB).

## Training Runs

### Run v1: 4 LoRA layers, 500 iterations
- **Config:** 4 layers (17% of model), rank 8, lr 1e-5, no dropout, max_seq 512
- **Data:** 1,240 train / 155 val / 155 test (1,550 total)
- **Training time:** ~10 min
- **Best val loss:** 0.493 (iter 500)
- **Adapter size:** 5.3 MB
- **Result (45 cases):** 57.8% → 80.0% (+22.2 pts)

### Run v3: 8 LoRA layers, 1000 iterations
- **Config:** 8 layers (33% of model), rank 8, lr 1e-5, no dropout, max_seq 256
- **Data:** Same 1,550 examples
- **Training time:** ~17 min (auto-resumed through GPU crashes)
- **Best val loss:** 0.414 (iter 1000)
- **Adapter size:** 11.2 MB
- **Result (45 cases):** 57.8% → 82.2% (+24.4 pts)

### Run v4: 8 LoRA layers, 4000 iterations — OVERFITTING
- **Config:** Same as v3, continued training from v3 checkpoint
- **Data:** Same 1,550 examples
- **Training time:** ~6 hours (massive crash/resume overhead)
- **Train loss at start:** 0.037 (already memorized)
- **Result (45 cases):** 73.3% — **regressed 8.9 pts from v3**
- **Lesson:** Model memorized data by iter 1000. More training on same data hurts.

### Run v5: 8 LoRA layers, 2000 iterations — BEST MODEL ★
- **Config:** 8 layers, rank 8, lr 1e-5 with cosine decay + warmup, **dropout 0.05**
- **Data:** 2,177 train / 272 val / 273 test (2,722 total, ~2x original)
- **Training time:** 28 minutes, zero crashes (AGX fix)
- **Final val loss:** 0.716 (still dropping)
- **Final train loss:** 0.439 (not memorized — dropout working)
- **Result (65 cases):** 60.0% → **81.5%** (+21.5 pts)
- **Result (45 cases):** 57.8% → **82.2%** (+24.4 pts)

### Run v6: Fresh from scratch, 2500 iterations
- **Config:** 8 layers, rank 8, dropout 0.05, cosine decay, 2500 iters
- **Data:** 2,490 train / 311 val / 312 test (3,113 total)
- **Training time:** 33 minutes, zero crashes
- **Final val loss:** 0.597 (lower than v5)
- **Result (65 cases):** 73.8% — lower than v5 despite more data
- **Lesson:** Aggressive safety training data confused the model. v5's cleaner dataset worked better.

## Key Discovery: macOS Tahoe GPU Watchdog Bug

### The Problem
Training on macOS 26.x (Tahoe) repeatedly crashes with `kIOGPUCommandBufferCallbackErrorImpactingInteractivity`. The macOS Metal GPU watchdog kills MLX training because command buffers block WindowServer display compositing.

### Root Cause
macOS Tahoe's "Liquid Glass" design language dramatically increased WindowServer GPU load. Apple Silicon shares one GPU between UI rendering and compute. Training with longer sequences (~256 tokens, ~0.8 it/sec) keeps the GPU busy long enough to trigger the watchdog.

### What We Tested

| Test | Display | Background | Result |
|---|---|---|---|
| 1 | Open | Minimal | Crashed iter ~180 |
| 2 | Open | Normal OS | Crashed iter ~53 |
| 3 | Open | Normal OS | Crashed iter ~43 |
| 4 (macOS 26.3.1) | Open | Normal OS | Crashed iter ~1 |
| 5 | **Closed (lid)** | **caffeinate -s** | **Completed 500/500** |

### The Fix
`AGX_RELAX_CDM_CTXSTORE_TIMEOUT=1` — an undocumented Apple GPU driver env var provided by an MLX collaborator in response to our GitHub issue (#3267). Completely eliminates the crash with display active.

Filed as: [ml-explore/mlx#3267](https://github.com/ml-explore/mlx/issues/3267)

## Training Data

### Dataset Versions

| Version | Total | Train | Source |
|---|---|---|---|
| v1 | 1,550 | 1,240 | 8 Claude Sonnet agents |
| v2 | 2,722 | 2,177 | + 7 more Sonnet batches |
| v3 | 3,113 | 2,490 | + targeted + safety data |

### Key Findings on Data
- **Quality > Quantity:** v5 (2,177 train) outperformed v6 (2,490 train). The extra aggressive safety examples in v6 confused the model.
- **1 epoch is enough:** Model peaks at ~0.8-1.0 epochs. Training beyond that degrades performance.
- **Dropout prevents memorization:** v5 train loss 0.439 at iter 2000 vs v3 train loss 0.037 at iter 1000.
- **Safety via SFT doesn't work:** 125+ refusal examples in training data, model still runs `rm -rf *`.

| 5,000 (v4/4000) | Worst | 73.3% | — |

With 1,240 examples and no dropout, the model memorized the data by iter 1000. Every checkpoint after was worse. Adding dropout (0.05) and cosine LR decay in v5 fixed this.
